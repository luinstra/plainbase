Kubernetes

Cluster layout, namespaces, and deployment manifests for the production estate.

Cluster layout

Every environment runs an isolated cluster with three control-plane nodes behind an internal load balancer. Worker capacity differs per environment; production keeps headroom for a full availability-zone outage.

EnvironmentNamespaceWorkersRegion
Productionprod12us-east-1
Stagingstaging4us-east-1
Developmentdev2us-west-2

Namespaces

One namespace per environment, named for the environment it serves. Namespaces are created by Terraform — never kubectl create namespace by hand, or the next apply will fight you.

  • Application workloads live in the environment namespace.
  • Cluster-wide tooling (ingress, cert-manager, metrics) lives in platform.
  • Anything in default is considered a mistake and is swept nightly.

Deployments

The app ships as a single container image. The Deployment, Service, and Ingress are templated in the platform chart; an environment only overrides replica count and image tag.

Rolling deploys

Rollouts are gradual by default — see the deploy guide for the full rollout checklist. To scale the web tier on the fly:

# confirm current state, then scale the web deployment
kubectl -n prod get deploy/web
kubectl -n prod scale deploy/web --replicas=8

Rollbacks

Every apply is recorded in the rollout history. Roll back to the previous revision with a single command — no manifest editing required:

kubectl -n prod rollout undo deploy/web
kubectl -n prod rollout status deploy/web
Before a production rollback — page the on-call owner and link the incident. Control-plane and prod namespace changes require a maintainer review before merge.

Scaling

The cluster autoscaler adds nodes when pods go unschedulable, but it is deliberately conservative. For a planned traffic event, pre-scale rather than relying on reactive scaling:

  1. Raise the worker minimum in the environment's Terraform variables.
  2. Bump the Deployment replica floor so capacity is ready before traffic arrives.
  3. Watch kubectl top nodes and the monitoring dashboard through the ramp.

Networking

TLS terminates at the load balancer and traffic is plain HTTP inside the mesh. Service discovery and DNS conventions are documented separately under Networking.
